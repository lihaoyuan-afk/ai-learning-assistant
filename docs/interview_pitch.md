# 3 分钟项目介绍稿

> 面试时照着逻辑讲，不要背稿。每段控制在 30-40 秒。

---

## 第一段：背景与动机（30 秒）

"我做这个项目的出发点很实际——我在学习一些技术文档时，常常看完就忘，缺少一个主动检验和记忆巩固的工具。所以我决定自己造一个，同时把它作为系统学习 AI Agent 工程的实践载体。

项目的核心定位不是聊天机器人，而是一个学习操作系统：用户上传 PDF，系统能帮你问答、出题自测、追踪哪些知识点还没掌握、安排什么时候该复习。"

---

## 第二段：整体架构（40 秒）

"技术栈上，前端是 Next.js 15 + TypeScript，后端是 FastAPI + Python，AI 编排用的是 LangGraph。

整个 RAG 管线是这样的：用户上传 PDF 后，我先做语义分块——按段落边界切，不是固定字符截断，这样语义比较完整。然后用 nomic-embed-text 生成 768 维向量写入 Qdrant。问答时，先向量检索 15 个候选，再用 BM25 做二次重排，最终取 Top 5 给 LLM，回答必须带页码来源。

数据持久化用 SQLite 做开发，Docker Compose 里配的是 PostgreSQL，带 healthcheck 和服务依赖。"

---

## 第三段：Agent 设计——重点（60 秒）

"Agent 这块我觉得是这个项目最有意思的地方。

我用 LangGraph 实现了三条有向状态图，其中 Chat Graph 里加了一个 Reflection 节点。具体逻辑是：LLM 生成回答之后，Reflection 节点会检测这个回答是不是回退信号——比如字数过短，或者包含"没有找到"这类表达。如果是，就自动把 retrieval 的 limit 从 5 扩到 8，重新检索再回答一次。同时用 retry_count 控制最多重试 1 次，防止无限循环。

这个设计让我对 Agent 的状态管理有了很实际的感受——终止条件是第一公民，LangGraph 的状态 TypedDict 让每个节点的输入输出都非常清晰，也很好测试。

另外我还加了一个学习规划 Agent：读取用户的薄弱知识点和文档列表，让 LLM 生成优先级排序的学习计划。LLM JSON 解析失败时会降级到启发式排序，不中断用户流程。"

---

## 第四段：工程挑战（30 秒）

"遇到的最大挑战是小模型的 JSON 可靠性问题。我最初用的是 1.5B 参数的 deepseek-r1，它几乎无法稳定输出合法 JSON，导致 Quiz 生成经常失败。

解决方案分两层：第一层是模型层，换成 qwen2.5:7b 后 JSON 成功率从 0-1 题提升到 6/6；第二层是工程层，我写了 `_strip_think` 剥离 deepseek 的推理标签、`_extract_json` 从混合文本里提取 JSON 块、`response_format` 降级策略，以及多种 fallback 解析路径，让系统在模型不配合时也能优雅处理。"

---

## 第五段：结尾（20 秒）

"项目目前 88 个 pytest 测试全部通过，全离线运行，CI 可以直接跑。前端 Next.js 生产构建零错误。

这个项目让我理解了 AI 工程最核心的挑战不是调用 API，而是怎么在 LLM 输出不可信的前提下，设计出可靠的工程边界。"

---

## 可能的追问与回答

**Q: 为什么选 LangGraph 不用 LangChain 的 Chain？**
"Chain 是线性的，无法表达条件分支和循环。我的 Reflection 重试逻辑本质上是一个有状态的循环，LangGraph 的状态图让这个循环可追溯、可测试。每个节点独立，单元测试直接注入状态就行。"

**Q: BM25 重排有实际提升吗？**
"在精确关键词查询下有明显提升，比如查'mAP@0.5'这种专有指标，纯向量检索可能返回泛泛的性能讨论，BM25 会优先命中包含这个精确词的段落。弱点是 BM25 对同义词不敏感，所以权重只给 0.3，向量保持 0.7。"

**Q: 项目有什么局限性？**
"三个明确的局限：第一，单用户，没有认证；第二，Reflection 只重试一次，更复杂的 Agent 可以做查询改写后多次重试；第三，掌握度评分是简单的线性加减，没有时间衰减，真实的记忆遗忘曲线应该是指数衰退的。这些都是下一步可以迭代的方向。"

**Q: 你用 Claude Code 具体怎么配合开发的？**
"我把每个功能点拆成独立任务交给 Claude Code，但不是直接让它写完就用。比如 Reflection 节点，我先描述设计意图，它给出实现后我检查状态边界和终止条件，发现 mock LLM 返回的字符串太短会误触发重试，我让它修 conftest 而不是改业务逻辑——这个判断是我做的。大的架构决策、模块边界、测试策略都是我先想清楚再用 Claude Code 执行，而不是反过来。"
